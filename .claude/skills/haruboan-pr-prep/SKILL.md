# PR 준비 절차

다음 5단계를 순서대로 실행한다. 단계 사이에 사용자 확인을 요청하지 말고 끝까지 진행하되, 실패가 나면 그 단계에서 멈추고 사유를 보고한다.

## 1. 변경 범위 점검

- `git status`, `git diff origin/main...HEAD --stat` 으로 전체 변경 파일을 본다.
- 한 PR 한 주제 원칙에 위배되는 무관한 변경이 섞여 있으면 사용자에게 분리 제안.

## 2. 린트·타입 체크

- `npm run lint` 실행. 에러가 있으면 표로 정리.
- `npm run build` 로 TypeScript 타입 에러 확인.

## 3. 테스트

- 테스트 파일이 있으면 실행. 없으면 "테스트 미설정" 으로 표기하고 계속 진행.
- 새 기능에 대응되는 테스트 케이스가 추가됐는지 확인 (없으면 사용자에게 알림).

## 4. 컨벤션 검증

- 변경된 코드에 대해 `haruboan:code-reviewer` 서브에이전트를 호출해 CLAUDE.md 위반 표를 받는다.
- 위반이 있으면 수정 후 다시 2단계부터 반복.

## 5. PR 본문 초안

- 다음 형식으로 출력한다.

  ```
  ## 요약
  - 한 줄 요약 1~3개

  ## 변경 사항
  - 파일·모듈 단위 핵심 변경

  ## 테스트
  - 추가/수정된 테스트와 검증 방법

  ## 리스크/주의
  - 마이그레이션·호환성·롤백 관련 메모 (없으면 "없음")
  ```

- 사용자가 그대로 `gh pr create --body` 에 붙여넣을 수 있는 형태로 출력.

## 종료 조건

- 5단계까지 모두 성공해야 "PR 준비 완료" 를 선언한다.
- 어느 단계든 실패하면 해당 단계까지의 결과 + 다음 권장 액션 한 줄을 출력하고 종료한다.
